Pipelight

LLM-in-the-Loop · Lightweight CLI CI/CD · Rust

一个用 Rust 实现的轻量级 CI/CD 工具，通过 LLM 自动修复 实现 build → fix → retry 闭环。 替代 Jenkins / GitLab CI 等重量级方案，提供本地优先、快速、可离线使用的 CI/CD 体验。

Rust + tokio Docker (bollard) Claude Code / Trae Skill DAG (petgraph) serde_yaml YAML pipeline

1. 使用方法

1.1 安装

# 克隆仓库并编译
git clone git@github.com:amwtke/deeparch_cicd.git
cd deeparch_cicd
cargo build --release

# 安装到 PATH
cp target/release/pipelight ~/.cargo/bin/

1.2 基本工作流

# 1) 自动检测项目类型，生成 pipeline.yml
$ pipelight init -d /path/to/your/project

# 2) 查看执行计划（不实际运行）
$ pipelight run --dry-run

# 3) 运行流水线（JSON 模式供 LLM 解析）
$ pipelight run --output json

# 4) 重试失败的步骤
$ pipelight retry --run-id abc123 --step pmd --output json

2. 三角色分层架构

LLM (Claude Code / Trae)
编排决策者 — 调用 pipelight CLI • 解析 JSON • 按 action 分发回调 • 修复代码 / 生成配置 / 打印报告 • 决定 retry
实现：pipelight-run Skill (SKILL.md) + Skill Memory
↓ pipelight run --output json   ↑ JSON { status, steps[], on_failure }
Pipelight Binary (Rust CLI)
调度执行器 — 检测项目类型 • 生成 pipeline • DAG 拓扑排序 • 异常模式匹配 • 结构化 JSON 输出 • 不做任何智能决策
关键模块：detector → pipeline_builder → scheduler → executor → output
↓ Docker API (bollard)   ↑ stdout / stderr / exit_code
Docker Containers
隔离执行环境 — 每个 step 独立容器 • bind-mount /workspace • 执行 shell 命令 • 容器自动清理
镜像：maven:3.9 • gradle:8-jdk17 • rust:latest • node:20 • alpine/git

为什么分离三个角色？

原则说明好处
Pipelight 不做智能决策只执行 + 报告，不搜索文档、不修改代码可测试、可复现
LLM 不直接操作 Docker只通过 CLI 间接控制容器避免权限泄露
Docker 只是沙箱提供隔离和可复现性，不参与业务逻辑环境一致性
独立升级：换 LLM（Claude → GPT）只需改 skill；换容器运行时（Docker → Podman）只需改 executor；加新语言只需加 detector + pipeline_builder 策略。

2b. 代码架构分层

src/
  main.rs                           // 入口
  cli/mod.rs                        // clap 命令定义 + dispatch 逻辑
  run_state/mod.rs                  // RunState 持久化 (JSON 文件)
  ci/
    detector/                       // 策略模式：项目类型检测
      base/mod.rs                   //   ProjectType 枚举 (Maven/Gradle/Rust/Node/Vue/Python/Go)
      maven.rs / gradle.rs / ...    //   各语言检测策略
    pipeline_builder/               // 策略模式：Step 生成 (StepDef trait)
      base/                         //   公共 step: ping_pong / git_pull / git_diff / build / test
      maven/ / gradle/ / ...        //   特有 step: pmd / spotbugs / fmt-check
    parser/mod.rs                   // YAML 解析 (serde_yaml)
    scheduler/mod.rs                // DAG 拓扑排序 (petgraph)
    executor/mod.rs                 // Docker 交互 (bollard crate)
    callback/
      command.rs                    // CallbackCommand 枚举（发生了什么）
      action.rs                     // CallbackCommandAction 枚举（LLM 该做什么）
    output/
      json.rs / plain.rs / tty/     // 三种输出格式

ExceptionMapping 异常匹配机制

Pipelight 不是简单地根据 exit code 决定回调，而是通过 StepDef::match_exception() 分析 stdout/stderr 内容：

exit_code != 0
    |
    v
StepDef::match_exception(exit_code, stdout, stderr)
    |
    +-- stderr 含 "PIPELIGHT_CALLBACK:auto_gen_pmd_ruleset"
    |   -> exception_key = "ruleset_not_found" -> AutoGenPmdRuleset
    |
    +-- stderr 含 "Cannot load ruleset"
    |   -> exception_key = "ruleset_invalid" -> AutoGenPmdRuleset
    |
    +-- stdout 含 "PMD Total: N violations"
    |   -> exception_key = "pmd_violations" -> AutoFix
    |
    +-- stdout 含 "BUILD FAILURE" (Maven)
    |   -> exception_key = "compilation_failure" -> AutoFix
    |
    `-- 默认 -> Abort

同一个 step 在不同失败原因下触发不同的 CallbackCommand。

3. 三角色通信协议

3.1 on_failure JSON 结构

每个 step 执行完毕后，JSON 输出中的 on_failure 字段携带完整回调信息：

{
  "on_failure": {
    "exception_key": "pmd_violations",     // 异常分类（溯源）
    "command": "auto_fix",                 // CallbackCommand（发生了什么）
    "action": "retry",                     // CallbackCommandAction（LLM 该做什么 = 分发键）
    "max_retries": 9,
    "retries_remaining": 8,
    "context_paths": [                     // LLM 需要读取的上下文文件
      "pipelight-misc/pmd-report/pmd-result.xml",
      "src/main/java/**/*.java"
    ]
  }
}

3.2 协议翻译链

CallbackCommand (发生了什么) ⟶ Registry 翻译 ⟶ CallbackCommandAction (LLM 该做什么) ⟶ Skill 处理表查具体操作

command 是溯源用（哪个异常），action 才是分发键（做什么类别的事）。

3b. 回调命令映射表

CallbackCommandActionLLM 具体操作
auto_fixretry读 stderr + context_paths 源码，修复后 retry
auto_gen_pmd_rulesetretry搜索项目 PMD 规则，生成 pmd-ruleset.xml，retry
pingretry打印 "pong"，直接 retry（通信测试）
git_failskip无操作，pipelight 已标记 skipped
fail_and_skipskip前置条件缺失，自动 skip
runtime_errorruntime_errorPipeline 终止，报告错误
abortabort严重代码问题，Pipeline 终止
test_print_commandtest_print解析 JUnit XML，按模块聚合打印测试表格
pmd_print_commandpmd_print解析 PMD XML，打印分类违规统计表
spotbugs_print_commandspotbugs_print解析 SpotBugs XML，打印分类缺陷表
git_diff_commandgit_diff_report打印 unstaged / staged / unpushed 分组
强制规则：LLM 收到 JSON 后必须遍历所有 step 的 on_failure，不论 step status 是 success/failed，也不论 pipeline 整体 status。
原因：test_print 等打印型 action 常搭在 allow_failure: true 的 step 上，其 status 为 success。

4. Pipelight 三种运行模式

Mode 1: 初始执行 (Run)

pipelight run -f pipeline.yml --output json [--skip step1,step2] [--ping-pong] [--full-report-only]

流程：加载 YAML → DAG 拓扑排序 → 按 batch 执行 → 每个 step 在 Docker 中运行 → 收集结果 → 异常匹配 → 生成 on_failure → 输出 JSON

退出码：0=全部成功   1=可重试 (retryable)   2=不可恢复 (failed)

--full-report-only 模式

PMD 切换为全量扫描 + 报告模式：扫描整个项目（非增量），始终 exit 0，violations 不触发 auto_fix。适用于发版前全量质量报告。

Mode 2: 重试模式 (Retry)

pipelight retry --run-id <id> --step <name> --output json

流程：

  1. 从 ~/.pipelight/runs/<run_id>.json 加载上次 RunState
  2. 验证指定 step 是 Failed + retries_remaining > 0
  3. 递减 retries_remaining
  4. 重新执行该 step
  5. reresolve_on_failure：重新匹配异常（不沿用上次状态）
  6. 成功则级联执行被上游阻塞的后续 step
  7. 更新并保存 RunState

级联 Retry 的三种 Skipped 判定

Skipped 原因示例级联？
active: false（配置关闭）ping-pong 默认不激活不级联
执行后被 Skip 策略跳过git-pull 失败 → git_fail → skip不级联
被上游阻塞而未执行build 失败 → test/pmd 未执行级联

Mode 3: 检查模式

pipelight run --dry-run      # 显示执行计划，不实际运行
pipelight validate           # 校验 YAML 格式 + DAG 无环
pipelight status --run-id <id>  # 查询历史运行状态
pipelight list               # 列出所有 step 及依赖关系

不启动任何 Docker 容器，只做 YAML 解析、DAG 校验、历史查询。用于开发调试和 CI 前置检查。

5. Harness 工程模式

什么是 Harness？

Harness = 测试线束 / 编排驱动器。一个薄层编排程序控制被测系统的输入输出。

pipelight-run Skill (Harness)
定义在 SKILL.md — 运行在 LLM 内部
↓ 调用 ↑ 解析 JSON
pipelight Binary (被编排的工具)
Rust CLI — 只执行 + 报告，不做智能决策

Harness 侧职责 (Skill)

  1. 决定何时调用 pipelight init / run / retry
  2. 解析 JSON，遍历所有 step 的 on_failure
  3. 按 action 分发到对应处理逻辑
  4. 执行"智能"操作（搜索文档、修复代码、生成配置）
  5. 决定是否继续 retry 或放弃

被编排侧职责 (Binary)

  1. 检测项目类型
  2. 执行 Docker 容器
  3. 收集输出、匹配异常模式
  4. 以结构化 JSON 报告结果
  5. 绝不做任何"智能"决策

协议约束

pipelight 产出 CallbackCommand
    (语义：发生了什么事)
        |
        v  CallbackCommandRegistry 翻译
pipelight 产出 CallbackCommandAction
    (语义：LLM 该做什么类别的动作)
        |
        v  两者都写入 JSON on_failure 字段
LLM 收到后按 action 查 skill 表
        |
        v
LLM 执行对应动作
    (修代码 / 打印报告 / skip / retry)

vs 传统 CI/CD

维度Jenkins / GitLab CIPipelight + Harness
失败处理报告失败，等人工修自动分析+修复+retry
配置管理手写 CI 文件init 自动检测生成
运行环境CI 服务器本地 (Docker)
反馈延迟分钟级秒级
修复闭环人看报告→改代码→推送LLM 自动闭环
扩展语言改 CI 脚本加策略模式

6. 未来改进方向

并行执行

DAG 已识别可并行 batch，利用 tokio 并发执行同一 batch 内的多个 step

更多语言支持

扩展 C/C++ (CMake)、Swift、Kotlin Multiplatform 等 detector + builder 策略

远程执行

支持远程 Docker daemon、Kubernetes Job、SSH 执行模式

增量构建优化

利用 git-diff 变更文件列表，只重新编译/测试受影响模块，缩短反馈周期

多 LLM 后端

抽象为通用 LLM agent 协议，支持 GPT / Gemini / 本地模型

Web Dashboard

RunState JSON 已是结构化数据，加前端展示历史记录、趋势图、失败热点
